# OpenMVS Workspace

This folder contains a lightweight `TUM RGB-D -> OpenMVS` workflow.

## Scripts

### `openmvs/setup_openmvs.sh`

What it does:
- clones `OpenMVS` into `openmvs/OpenMVS/`
- downloads local third-party dependencies used by this workflow
- configures and builds OpenMVS

How to use:

```bash
bash openmvs/setup_openmvs.sh
```

### `openmvs/export_tum_to_openmvs.py`

What it does:
- reads a TUM RGB-D sequence from a config file
- selects synchronized RGB, depth, and pose frames
- exports `images/` and `colmap/sparse/` for OpenMVS
- writes `seed_from_depth.ply` as a quick RGB-D reference point cloud
- clears the existing workspace first by default

How to use:

```bash
python openmvs/export_tum_to_openmvs.py --config configs/rgbd/tum/fr3_office.yaml --workspace-name fr3_office_openmvs --frame-stride 10 --max-frames 120
```

### `openmvs/run_openmvs_tum.sh`

What it does:
- optionally re-exports the TUM sequence into the workspace
- runs `InterfaceCOLMAP`
- runs `DensifyPointCloud`
- removes intermediate `.dmap` files by default
- stores logs in `log/oppenmvs/<workspace>/`

How to use:

```bash
bash openmvs/run_openmvs_tum.sh --workspace-name fr3_office_openmvs --reexport --frame-stride 10 --max-frames 120
```

Keep `.dmap` files for debugging:

```bash
bash openmvs/run_openmvs_tum.sh --workspace-name fr3_office_openmvs --reexport --frame-stride 10 --max-frames 120 --keep-dmaps
```

## Default Naming

- default workspace name: `fr3_office_openmvs`
- default log folder: `log/oppenmvs/fr3_office_openmvs/`

## Common Results

- `openmvs/workspaces/<workspace>/images/`
  Exported RGB images used by OpenMVS.
- `openmvs/workspaces/<workspace>/colmap/sparse/`
  Exported cameras, images, and sparse metadata.
- `openmvs/workspaces/<workspace>/scene.mvs`
  OpenMVS scene file after `InterfaceCOLMAP`.
- `openmvs/workspaces/<workspace>/scene_dense.mvs`
  OpenMVS dense scene file after `DensifyPointCloud`.
- `openmvs/workspaces/<workspace>/scene_dense.ply`
  Dense point cloud generated by OpenMVS.
- `openmvs/workspaces/<workspace>/seed_from_depth.ply`
  Quick reference point cloud sampled directly from TUM depth maps.
- `openmvs/workspaces/<workspace>/depth*.dmap`
  Per-view depth maps generated by OpenMVS during dense reconstruction. These are removed by default unless `--keep-dmaps` is used.
- `log/oppenmvs/<workspace>/`
  Run logs.

## Output Explanation

- `seed_from_depth.ply`
  A simple point cloud created by directly back-projecting TUM depth images with known camera poses. This is fast and useful as a reference, but it is not the final OpenMVS dense reconstruction.
- `scene.mvs`
  The OpenMVS scene file created after `InterfaceCOLMAP`. It stores the camera setup and scene description that OpenMVS will use.
- `depth*.dmap`
  Intermediate depth maps estimated by OpenMVS for individual views. These are temporary reconstruction products and are mainly useful for debugging or inspecting how dense reconstruction behaves per image.
- `scene_dense.mvs`
  The OpenMVS scene file after densification. It records the dense reconstruction state.
- `scene_dense.ply`
  The main OpenMVS dense point cloud output. This is the file you usually want if your goal is a dense 3D point cloud from OpenMVS.

## Comparing in CloudCompare


We can compare:

- `openmvs/workspaces/<workspace>/seed_from_depth.ply`
  Raw RGB-D back-projected point cloud.
- `openmvs/workspaces/<workspace>/scene_dense.ply`
  OpenMVS dense point cloud.
- `3d_construction/outputs/fr3_office.pcd`
  TSDF point cloud, if you also ran the TSDF pipeline.

 

What to look for:

- `seed_from_depth.ply` is usually noisier and contains more raw depth artifacts.
- `scene_dense.ply` is the OpenMVS multi-view dense reconstruction result.
- `fr3_office.pcd` from TSDF is usually smoother because it fuses depth into a volumetric representation.

## Notes

- TUM RGB-D is an RGB-D dataset, not a pure image-only SfM dataset.
- Smaller `frame-stride` means more images and longer runtime.
- Smaller `resolution-level` means higher image resolution and longer runtime.
- Larger `number-views`, `iters`, and `geometric-iters` usually increase runtime.
- `seed_from_depth.ply` is a fallback reference result and is not the final OpenMVS dense cloud.
- This workflow keeps results only in the workspace to avoid duplicate large files.
- `.dmap` files are intermediate files and are removed by default to save space.
